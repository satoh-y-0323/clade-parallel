# /agent-mcp-setup コマンド

MCPセットアップエージェント（mcp-setup）を起動する。親 Claude がユーザーとの Q&A（ヒアリング）を担当し、完了後にサブエージェントを一発起動してMCP接続設定とスキルファイルを生成する。

## 親 Claude の責務

このコマンドは親 Claude が直接実行する。サブエージェントはMCP追加処理とスキルファイル生成のみを担当する。

## 実行フロー

### Step 1: 上流レポートの読み込み（必要に応じて）

既存のMCPスキルファイルを確認したい場合は Glob で `.claude/skills/project/*-mcp.md` を検索する。
通常はスキップして Step 2 へ進む。

### Step 2: Q&A の実施

以下のヒアリングを順番に行う（1問ずつテキストで出力し、回答を受けてから次へ進む）。

**Step 2-1: MCPサーバの種別確認**

```
どのようなMCPサーバを追加したいですか？

1. 公開MCPサーバ（npm / GitHub で公開されているもの）
   例: GitHub MCP, Slack MCP, Notion MCP, Postgres MCP など
2. プライベート・社内MCPサーバ（社内で開発・運用されているもの）
   例: 社内API MCPサーバ, 独自ツールのMCPラッパー など

番号またはサーバ名をお知らせください。
```

**（公開MCPサーバの場合）Step 2-2a: 接続情報の確認**

回答を受けたら、サーバの情報を整理してユーザーに提示し確認を得る:

```
【{サーバ名}】について確認しました。

- パッケージ: {npm パッケージ名}
- 起動コマンド: {コマンド}
- 必要な環境変数: {リスト}
- 使えるツール: {ツール一覧}

このプロジェクト専用（project スコープ）として追加します。
追加してよいですか？ [yes / no]
※ 全プロジェクトで使いたくなった場合は後から /promote で昇格できます。
```

**（プライベートMCPサーバの場合）Step 2-2b〜2-6: 接続情報のヒアリング**

```
Step 2-2b: サーバの名前（識別子）を教えてください。
   例: my-company-api, internal-db など
```
↓ 回答を受けてから次へ
```
Step 2-3b: 接続方式を教えてください。
   - stdio: ローカルプロセスを起動して接続（コマンド実行型）
   - sse / http: URLに接続（リモートサーバ型）
```
↓ 回答を受けてから次へ

（stdio の場合）
```
Step 2-4b: 起動コマンドを教えてください。
   例: node /path/to/server.js
       python -m my_mcp_server
       npx my-company-mcp-server
```

（sse / http の場合）
```
Step 2-4b: サーバのURLを教えてください。
   例: http://internal.example.com:3000
```
↓ 回答を受けてから次へ
```
Step 2-5b: 認証・環境変数は必要ですか？
   必要な場合はキー名と説明を教えてください。
   例: API_KEY（社内APIの認証キー）, BASE_URL（サーバのベースURL）
```
↓ 回答を受けてから次へ
```
Step 2-6b: このMCPサーバで何ができるか教えてください。
   提供されるツール・機能を簡単に説明してもらえるとスキルファイルに記載できます。
```

**Step 2-7: 設定内容の確認**

ヒアリング内容を整理してユーザーに提示し確認を得る:

```
以下の設定を追加します。よいですか？

サーバ名: {name}
スコープ: project（このプロジェクト専用）
コマンド: {command}  または  URL: {url}
環境変数: {env vars}

[yes / no]
```

### Step 3: Q&A 結果の整理

ユーザーの回答を以下の構造に整理する:
- MCPサーバ種別（公開 / プライベート）
- サーバ名
- 接続方式（stdio / sse / http）
- 接続情報（コマンドまたはURL）
- 必要な環境変数
- 提供されるツール・機能の説明

### Step 4: サブエージェントの一発起動

Agent ツールで `subagent_type: mcp-setup` を指定して起動する。プロンプトに以下を含める:

```
## 作業依頼
MCPサーバの接続設定追加とスキルファイル生成

## ユーザーとの Q&A 結果

### MCPサーバ種別
{公開 / プライベート}

### サーバ情報
- サーバ名: {name}
- 接続方式: {stdio / sse / http}
- コマンド: {command（stdioの場合）}
- URL: {url（sse/httpの場合）}
- 必要な環境変数: {リスト またはなし}
- 提供されるツール・機能: {説明}

## 出力指示
1. `claude mcp add` コマンドでMCPサーバを追加すること
2. スキルファイルを `.claude/skills/project/{name}-mcp.md` に生成すること（Write ツール使用可）
3. AskUserQuestion / SendMessage は使わないこと
4. セットアップ完了後は終了すること（完了報告は親 Claude が担当）
5. 最終メッセージに接続設定状況とスキルファイルパスを含めること
```

否認後の再生成時はプロンプトに以下を追加する:
```
## 再生成モード
- ユーザーからの修正指示: {指示内容}
```

### Step 5: 結果の受け取り

サブエージェントの最終出力から接続設定状況とスキルファイルパスを確認する。

### Step 6: 承認確認

ユーザーに以下をテキストで提示する:

```
MCPサーバのセットアップが完了しました。

接続設定: project スコープに追加済み
スキルファイル: {ファイルパス}

{環境変数の設定が必要な場合はその手順}

内容を確認して、このセットアップでよいですか？（yes / no）
修正が必要な場合はその内容をお知らせください。

全プロジェクトで使いたい場合は /promote でグローバルに昇格できます。
```

### Step 7: 承認記録

mcp-setup は固定のレポートファイルがないため、承認記録は省略する。

### Step 8: 否認時の再起動

否認の場合、修正指示を含めた新プロンプトで Step 4 から繰り返す。

---

## 用途
- 公開MCPサーバ（npm / GitHub）の調査・接続設定・スキルファイル生成
- プライベート・社内MCPサーバの手動設定・スキルファイル生成
- 既存MCPサーバのスキルファイルのみ作成（接続設定済みの場合）

## 注意
- 標準ワークフロー（フェーズ構成）とは独立して動作する
- このコマンドは単独で完結する（他エージェントへの連携は不要）
- 環境変数の値（APIキー等）はファイルに直接書かず .env 管理を推奨する
