# /agent-doc-writer コマンド

ドキュメントライターエージェント（doc-writer）を起動する。親 Claude がユーザーとの Q&A を担当し、完了後にサブエージェントを一発起動してドキュメントを生成する。

## 親 Claude の責務

このコマンドは親 Claude が直接実行する。サブエージェントは Q&A 完了後に一発起動する。

## 実行フロー

### Step 1: 上流レポートの読み込み（必要に応じて）

必要に応じて関連する既存ファイル・ドキュメントを確認する。
通常は Q&A でユーザーから対象を特定するのでスキップして Step 2 へ進む。

### Step 2: Q&A の実施

以下の質問を順番にテキストで出力してユーザーの回答を待つ（1問ずつ出力し、回答を受けてから次へ進む）。

**Q1: 何を作りたいか**

```
どんなドキュメントを作成しますか？

  [mermaid]   Mermaid 図（フロー図・クラス図・ER図・シーケンス図など）
  [readme]    README（プロジェクト概要・セットアップ手順・使い方）
  [manual]    操作手順書・運用手順書（画面操作・コマンド手順など）
  [api]       API仕様書（エンドポイント・リクエスト/レスポンス定義）
  [spec]      仕様書（既存コードからの逆引き仕様まとめ）
  [other]     その他（自由記述）
```

**Q2: 対象ファイル・ディレクトリ**

```
対象となるファイルまたはディレクトリを教えてください。
（例: src/batch/HogeProcessor.cs、src/services/ 配下全体、など）
```

回答を受けたら、Glob/Grep/Read で対象を確認する。
対象が広い場合は「〇〇ファイルが見つかりました。全て対象でよいですか？」と確認する。

**Q3: 読み手**

```
このドキュメントを読む人は誰ですか？

  [dev-new]   開発チームの新メンバー（コードを読み始める人）
  [dev-team]  開発チーム内（既存メンバーへの共有・引き継ぎ）
  [ops]       運用・保守担当（エンジニアだがコードを書かない）
  [biz]       業務担当・非エンジニア（システムの利用者・管理者）
  [external]  外部レビュアー・顧客・発注者
  [other]     その他（自由記述）
```

**Q4: 目的**

```
このドキュメントを作る目的を教えてください。

  [overview]  全体把握・初見理解のため
  [handover]  引き継ぎ・担当交代のため
  [review]    レビュー・承認を得るため
  [trouble]   障害時の調査・対応のため
  [onboard]   新メンバーのオンボーディングのため
  [other]     その他（自由記述）
```

**Q5: 粒度**（Mermaid 図の場合は必須、他は任意）

Mermaid 図を選択した場合、または「どこまで詳しく書くか」が重要な場合に質問する:

```
どのレベルの粒度で記述しますか？

  [high]    高レベル（モジュール・サービス単位。全体の流れを一目で把握できる）
  [mid]     中レベル（クラス・関数単位。主要な処理の繋がりが分かる）
  [low]     低レベル（メソッド・フィールド単位。詳細な実装が分かる）
```

**Q6: 出力先の確認**

```
ドキュメントをどこに保存しますか？

  [reports]  .claude/reports/doc-{名前}.md（一時保管・レポート扱い）
  [project]  プロジェクト内の指定パス（例: docs/architecture.md）
  [show]     ファイルに保存せず、ここに表示するだけ
```

**Q7: 再確認**

```
以下の内容でドキュメントを作成します。よいですか？

・種別: {Q1の回答}
・対象: {Q2の回答}
・読み手: {Q3の回答}
・目的: {Q4の回答}
・粒度: {Q5の回答（該当する場合）}
・出力先: {Q6の回答}

問題なければ「yes」、変更がある場合はその内容をお知らせください。
```

### Step 3: Q&A 結果の整理

ユーザーの回答を以下の構造に整理する:
- ドキュメント種別
- 対象ファイル・ディレクトリ（Glob で確認した実際のパスも含む）
- 読み手
- 目的
- 粒度
- 出力先パス

### Step 4: サブエージェントの一発起動

Agent ツールで `subagent_type: doc-writer` を指定して起動する。プロンプトに以下を含める:

```
## 作業依頼
ドキュメントの生成

## ユーザーとの Q&A 結果

### Q1: ドキュメント種別
A: {回答}

### Q2: 対象ファイル・ディレクトリ
A: {回答}
確認済みファイル一覧: {Glob で確認したファイル一覧}

### Q3: 読み手
A: {回答}

### Q4: 目的
A: {回答}

### Q5: 粒度
A: {回答（該当する場合）}

### Q6: 出力先
A: {回答}
出力先パス: {具体的なパス}

## 出力指示
- 出力先: {Q6 で指定されたパス}（Write ツールで直接書き込む）
- [show] が指定された場合はファイル保存せず、最終メッセージにドキュメント本文を含めること
- AskUserQuestion / SendMessage は使わないこと
- ドキュメント生成後は終了すること（確認は親 Claude が担当）
- 最終メッセージに生成したファイルパス（または「表示のみ」）を含めること
```

否認後の再生成時はプロンプトに以下を追加する:
```
## 再生成モード
- ユーザーからの修正指示: {指示内容}
```

### Step 5: 結果の受け取り

サブエージェントの最終出力から生成されたファイルパスを確認する。
`[show]` の場合はドキュメント本文を確認する。

### Step 6: 承認確認

ユーザーに以下をテキストで提示する:

```
ドキュメントを生成しました。

保存先: {パス または「表示のみ」}
種別:   {種別}
対象:   {対象ファイル・ディレクトリ}

修正が必要な箇所はありますか？
  [ok]     このまま完了する
  [fix]    修正したい（修正内容を教えてください）
  [redo]   粒度・対象・構成を変えてやり直す
```

### Step 7: 承認記録

doc-writer はレポートファイルを生成しない場合もあるため、承認記録は省略可。
`.claude/reports/doc-*.md` に保存した場合は、シェルインジェクション対策としてコメントを tmp ファイル経由で渡す:

1. `node .claude/hooks/clear-tmp-file.js --path .claude/tmp/approval-comment.md` を実行
2. Write ツールで `.claude/tmp/approval-comment.md` にユーザーの承認コメントを書き込む（コメントなしの場合は空文字列）
3. 以下を実行:

```bash
node .claude/hooks/record-approval.js {ファイル名} {yes|no} doc --comment-file .claude/tmp/approval-comment.md
```

### Step 8: 否認時の再起動

`fix` または `redo` の場合、修正指示を含めた新プロンプトで Step 4 から繰り返す。

---

## 用途
- Mermaid 図の生成（フロー図・クラス図・ER図・シーケンス図）
- README の作成・更新
- 操作手順書・運用手順書の作成
- API仕様書の作成
- 既存コードからの仕様書逆引き生成

## 注意
- 標準ワークフロー（フェーズ構成）とは独立して動作する
- このコマンドは単独で完結する（他エージェントへの連携は不要）
- ソースコードの変更は行わない
