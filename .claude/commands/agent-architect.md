# /agent-architect コマンド

設計・アーキテクチャエージェント（architect）を起動する。親 Claude がユーザーとの Q&A を担当し、完了後にサブエージェントを一発起動してレポートを生成する。

## 親 Claude の責務

このコマンドは親 Claude が直接実行する。サブエージェントは Q&A 完了後に一発起動する。

## 実行フロー

### Step 1: 上流レポートの読み込み

Glob で `.claude/reports/requirements-report-*.md` を検索し、最新のファイルを Read する。
requirements-report が存在しない場合はスキップし、Q&A で要件を直接確認する。

### Step 2: Q&A の実施

requirements-report の有無に関わらず**必ず Q&A を実施する**。

architect の本来の役割は「要件を設計観点で掘り下げること」であるため、requirements-report がある場合も Q&A をスキップしてはならない。

以下の質問を順番にテキストで出力してユーザーの回答を待つ（1問ずつ出力し、回答を受けてから次へ進む）。

**requirements-report が存在する場合の Q&A:**

**Q1: 設計スコープの確認**

```
要件定義レポートを確認しました。設計を進める前に確認させてください。

今回の設計で扱う範囲（スコープ）を確認します。
・要件レポート全体を設計対象としますか？
・それとも特定の機能・コンポーネントに絞りますか？

（例: 全体設計 / 認証基盤のみ / APIレイヤーのみ など）
```

**Q2: 深堀りしたい点の確認**

```
設計で特に検討してほしいトレードオフや技術選定はありますか？

{requirements-report の「architectへの引き継ぎ事項」がある場合はその内容を踏まえて質問を組み立てる}

例:
・パフォーマンス重視 vs 開発速度重視
・モノリシック構成 vs マイクロサービス構成
・データ整合性 vs 可用性 など
```

**Q3: 制約・優先度の確認**

```
設計上の制約や優先度について確認させてください。

・今回特に重視する品質特性（パフォーマンス・セキュリティ・保守性など）はありますか？
・既存の技術スタック・ライブラリで維持すべきものはありますか？
```

**requirements-report が存在しない場合の Q&A:**

**Q1: 設計対象の確認**

```
どのような機能・システムを設計しますか？
設計の対象と背景を教えてください。
```

**Q2: 技術選定・トレードオフの確認**

```
設計で特に検討したいトレードオフや技術選定はありますか？

例:
・パフォーマンス重視 vs 開発速度重視
・モノリシック構成 vs マイクロサービス構成 など
```

**Q3: 制約・優先度の確認**

```
設計上の制約や優先度について確認させてください。

・今回特に重視する品質特性（パフォーマンス・セキュリティ・保守性など）はありますか？
・既存の技術スタック・ライブラリで維持すべきものはありますか？
```

### Step 3: Q&A 結果の整理

ユーザーの回答を以下の構造に整理する:
- 確認済みの技術要件・制約
- トレードオフの選択結果
- 優先品質特性
- 設計の前提条件

### Step 4: サブエージェントの一発起動

Agent ツールで `subagent_type: architect` を指定して起動する。プロンプトに以下を含める:

```
## 作業依頼
アーキテクチャ設計レポートの作成

## 上流レポートのパス
- requirements-report: {パス または「なし」}

## ユーザーとの Q&A 結果

### Q1: 深堀り確認への回答
A: {回答}

### Q2: トレードオフの選択
A: {回答}

### Q3: 制約・優先度
A: {回答}

## 出力指示
- 出力先: `.claude/reports/architecture-report-*.md`（write-report.js 経由）
- 最終メッセージにレポートファイルパスを必ず含めること（形式: `ファイル: .claude/reports/architecture-report-YYYYMMDD-HHmmss.md`）
- AskUserQuestion / SendMessage は使わないこと
- レポート生成後は終了すること（承認確認は親 Claude が担当）
```

否認後の再生成時はプロンプトに以下を追加する:
```
## 再生成モード
- 前回レポート: {前回レポートパス}
- ユーザーからの修正指示: {指示内容}
```

### Step 5: レポートパスの受け取り

サブエージェントの最終出力から正規表現 `.claude/reports/architecture-report-\d{8}-\d{6}\.md` でレポートファイルパスを抽出する。

### Step 6: 承認確認

ユーザーに以下をテキストで提示する:

```
アーキテクチャ設計レポートを `{ファイルパス}` に保存しました。内容を確認して、この設計を承認しますか？（yes / no）
修正が必要な場合はその内容もお知らせください。
```

### Step 7: 承認記録

シェルインジェクション対策としてコメントは tmp ファイル経由で渡す:

1. `node .claude/hooks/clear-tmp-file.js --path .claude/tmp/approval-comment.md` を実行
2. Write ツールで `.claude/tmp/approval-comment.md` にユーザーの承認コメントを書き込む（コメントなしの場合は空文字列）
3. 以下を実行:

```bash
node .claude/hooks/record-approval.js {ファイル名} {yes|no} architecture --comment-file .claude/tmp/approval-comment.md
```

### Step 8: 否認時の再起動

否認の場合、修正指示と前レポートパスを含めた新プロンプトで Step 4 から繰り返す。

---

## 用途
- システム設計・アーキテクチャ決定・技術選定
- ADR（Architecture Decision Record）の作成
- アーキテクチャ設計レポート（`architecture-report-*.md`）の作成

## 注意
- 作業開始前に最新の `requirements-report-*.md` を確認する（Step 1）
