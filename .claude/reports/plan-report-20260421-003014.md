## developer への注意事項

**F1 実装（Popen 切替）時の最重要ポイント**
- `subprocess.Popen(cmd, cwd=task.cwd, env=env, stdout=subprocess.PIPE, stderr=subprocess.PIPE, text=True)` を使用（`check=True` 相当の振る舞いは `returncode` を `TaskResult.ok` プロパティで評価するため不要）
- `try: stdout, stderr = proc.communicate(timeout=task.timeout_sec)` で通常パス、`except subprocess.TimeoutExpired:` で `proc.kill()` → `proc.communicate()`（後者は引数なし = kill 後のバッファ flush 用）
- `duration_sec = time.perf_counter() - start` は finally か両パスで必ず計算
- `FileNotFoundError`（`claude` 未検出）は Popen 起動時点で即 raise されるため try の外側でキャッチして `RunnerError` に変換する実装方針を維持
- text=True により stdout/stderr は `str` で返る（bytes 両対応デコード処理は削除可能 = コード簡素化）

**F2 実装時の最重要ポイント**
- `Future.exception()` は `BaseException | None` を返す。`None` は正常完了、`RunnerError` は最初の 1 件のみ保持、それ以外の `BaseException`（KeyboardInterrupt / SystemExit）は再送出
- `runner_error` 変数のスコープは `for future in futures:` ループの外側で初期化、ループ内で最初の 1 件のみ代入
- コメントを「Should not happen」から「BaseException 系（KeyboardInterrupt 等）のみここに到達する」に書き換える

**F3 実装時の最重要ポイント**
- ブロックリストは **frozenset** として `manifest.py` のモジュールレベルに定義（再計算コスト削減 + 不変性担保）
- 対象キー: `LD_PRELOAD`, `LD_LIBRARY_PATH`, `LD_AUDIT`, `DYLD_INSERT_LIBRARIES`, `DYLD_LIBRARY_PATH`, `PYTHONPATH`（T11 の推奨セット）
- キー判定は大小文字を区別しない方が安全（Linux の env var は case-sensitive だが、安全側に寄せるなら `.upper()` 比較）。ただし既存テスト（`FOO` / `MY_VAR` を許容）と互換性を保つため、まずは case-sensitive 完全一致で実装し、必要に応じて後で厳格化する

**R4 実装（CladeParallelError）時の最重要ポイント**
- `src/clade_parallel/_exceptions.py` を新設し `class CladeParallelError(Exception): """Base exception for all clade-parallel errors."""` のみを置く
- `manifest.py` / `runner.py` の先頭で `from ._exceptions import CladeParallelError` を追加し、`class ManifestError(CladeParallelError)` / `class RunnerError(CladeParallelError)` に変更
- `__init__.py` で `from .manifest import ...` に加え `from ._exceptions import CladeParallelError` を追加して公開 API に含める
- **後方互換**: 既存の `except ManifestError:` / `except RunnerError:` は継続して動作する（継承関係なので）

**コーディング規約の継続遵守**
- Black / Ruff 行長 88、ダブルクォート
- `pathlib.Path` 統一・`os.path` 文字列操作禁止は引き続き厳守
- 全 public API に型注釈（`X | None` 構文）と Google Style docstring
- `frozen=True` dataclass は変更禁止（既存設計を壊さない）
- **Clade モジュールは絶対に import しない**（前サイクルから継続）

**承認履歴から読み取った傾向**
- 全レポート承認時にコメントが付いており、特に T10/T11 の承認コメントに「今日中に修正サイクルを回す方針」と明記 → スピード優先で進める
- 否認履歴なし → 減点要素のない実装を継続すれば承認率は維持される見込み
