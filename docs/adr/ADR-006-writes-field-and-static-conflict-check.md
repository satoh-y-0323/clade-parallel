# ADR-006: `writes:` フィールドと静的衝突検査の導入

- **Date:** 2026-04-21
- **Status:** Accepted (partially superseded by ADR-011)

## Context

clade-parallel v0.2 において、各タスクが書き込むファイルパスを事前に宣言する `writes:` フィールドを導入する。
複数のタスクが同一のファイルに並列書き込みを行うと、データ破損や予期しない上書きが発生しうる。
この問題を実行前に検知するため、マニフェストのロード時に静的衝突検査を行う仕組みが必要である。

本 ADR は以下の設計判断を記録する:

1. `Task.writes` フィールドの型と保管形式
2. パス正規化の戦略
3. `writes:` フィールドを受理するバージョン範囲
4. 衝突検査の実施タイミングとエラー報告方式

また、ADR-005（`clade_plan_version` によるバージョンゲート）と一部の前提が変わるため、その関係を明示する。

## Decision

### 1. `Task.writes` は `tuple[str, ...]`（絶対 POSIX パス文字列）として保持する

`Task` dataclass は `frozen=True` であり、すべてのフィールドはハッシュ可能でなければならない。
ミュータブルな `list` は `__hash__` を破壊するため採用しない。
要素の型は `Path` ではなく `str`（解決済み絶対 POSIX パス文字列）とする。
これにより `WindowsPath` / `PosixPath` の分岐を排除し、エラーメッセージ出力での追加変換が不要になる。
デフォルト値は不変オブジェクトの空タプル `()` とし、`writes:` を省略した v0.1 マニフェストとの後方互換を保つ。

### 2. 正規化は `Path.resolve().as_posix()` のみ行い、casefold・OS 固有正規化は行わない

相対パスはタスクの `cwd`（`_parse_task()` 内で既に解決済みの絶対パス）を基準に解決する。
`Path.resolve()` を用いることで `..` セグメントの解決・シンボリックリンクの解消・プラットフォーム固有の正規化を標準ライブラリに委ねる。
`as_posix()` でパス区切り文字を `/` に統一し、Windows / macOS / Linux 間の比較結果を安定させる。
`os.path.normcase` による大小文字の独自 casefold は行わない。
理由: 静的チェックの目的は「同一マニフェスト内で意図的に同じパスを宣言した場合の検知」であり、
`Foo.txt` と `foo.txt` のような表記ゆれへの対処は YAGNI と判断した（OS によって case-sensitive の扱いも異なるため）。

### 3. `writes:` は v0.1 / v0.2 両バージョンのマニフェストで受理する

要件が「バージョン問わずオプション扱い」と明示しているため、`clade_plan_version: "0.1"` のマニフェストに
`writes:` が記載されていても正常に処理する。
これは ADR-005 の「未知キーは無視する」という前方互換ポリシーの延長ではなく、
`writes:` を v0.2 の主機能として**明示的にサポートする**決定である（下記「ADR-005 との関係」参照）。

### 4. 衝突検査は `load_manifest()` 内で全タスクパース後に実施し、全衝突を 1 回のエラーで報告する

実装上は `_check_writes_conflicts(tasks)` というプライベートヘルパを新設し、
全タスクのパース完了後・`Manifest` インスタンス生成前に呼び出す。
衝突が 1 件でも検出された場合は `ManifestError` を送出する。
最初の衝突で即座に打ち切るのではなく、全衝突を集約してから 1 回のエラーとして報告する。
これによりユーザーはマニフェストを 1 度の修正で直すことができる。
エラーメッセージにはどのパスがどのタスクによって宣言されているかを列挙し、パスをソートして決定論的にする。

## ADR-005 との関係

ADR-005 は「未知キー（`writes:`, `depends_on:` 等）はパーサが静かに無視する」と記述し、
「`clade_plan_version: "0.2"` のマニフェストを v0.1 ツールで読むと、バージョンゲートで弾かれる前に
未知キーが問題になることはない」という前提に立っていた。

本 ADR（ADR-006）は `writes:` を v0.1 / v0.2 の両方で処理対象にするため、
ADR-005 の記述と実装の間に差異が生じる。整理すると以下のとおり:

- **ADR-005 は廃止しない**: バージョンゲートの仕組み（`SUPPORTED_PLAN_VERSIONS` による照合）は引き続き有効である。
- **`depends_on:` 等の将来フィールドについては ADR-005 の「未知キーは無視する」が依然有効**:
  まだ実装されていないフィールドは従来どおり静かに無視される。
- **`writes:` のみが例外的に明示サポートに昇格した**: ADR-006 によって `writes:` は v0.2 の主機能として
  v0.1 / v0.2 両バージョンで受理・処理されることが明確に定義された。

## Alternatives Considered

**`Task.writes` を `list[str]` にする**: ミュータブルなため `frozen=True` のハッシュ可能性を破壊する。採用しない。

**`Task.writes` を `frozenset[str]` にする**: 自動重複除去とハッシュ可能性を持つが、
順序が保持されないため YAML の記述順とエラーメッセージの再現性が損なわれる。採用しない。

**`Task.writes` を `tuple[Path, ...]` にする**: 型として強いが、`WindowsPath` / `PosixPath` の差異が
テストを不安定にし、エラーメッセージ出力時に追加変換が必要になる。採用しない。

**`Path.resolve()` に加えて `os.path.normcase` で casefold する**:
Windows / macOS における大小文字無視の衝突検知が強化されるが、
Linux（case-sensitive）との挙動差が増え、実行前の静的チェックの目的に対してオーバーエンジニアリングとなる。採用しない。

**v0.1 マニフェストで `writes:` が書かれた場合に警告を出す**:
ユーザー体験は改善されるが、ログ機構の導入・テストコスト増を招く。
要件がシンプルな受理を求めているため採用しない。

**最初の衝突で即座に `ManifestError` を送出する**:
実装が最短になるが、ユーザーが衝突を 1 件修正するたびに再実行を繰り返す必要がある。
全衝突を集約して 1 回のエラーにする方がユーザー体験が大幅に改善されるため採用しない。

## Consequences

- `manifest.py` に `_normalize_write_path` と `_check_writes_conflicts` の 2 つのプライベートヘルパを追加する
- `Task` dataclass に `writes: tuple[str, ...] = ()` フィールドを追加する（後方互換・既存コードの破壊なし）
- `SUPPORTED_PLAN_VERSIONS` を `frozenset({"0.1", "0.2"})` に拡張する
- `writes:` の衝突エラーは `ManifestError` で統一し、新しい例外クラスは導入しない
- `runner.py` は変更しない（`writes:` は静的チェック専用であり、実行時の書き込み監視は v0.3 worktree 隔離で対応する）
- 将来 `depends_on:` 等の新フィールドを追加する際は、同様に本 ADR に倣って ADR を新規作成して方針を記録すること
- `_check_writes_conflicts` はパース時点での `Path.resolve()` 結果で比較するため、エージェントが実行時にシンボリックリンクを差し替えた場合の衝突は検知できない（static check の固有の制約）
- **[ADR-011 により部分的に更新]** `_normalize_write_path` が返すパスの意味付けが変更された。v0.5 以降、`Task.writes` は `Path.resolve()` 済みパスではなく、`os.path.normpath` で `..` を解決したユーザー宣言パス（symlink 展開前）を保持する。詳細は ADR-011 を参照。
